## Key Findings

### API Correctness — StateView + @view()
- **Two-tier visibility:** Element-level `view.add/remove` is sufficient
- **@view() optimization:** Skip field-level decorators in Phase 1. Simplifies schema mutations.
- **No new Colyseus APIs needed:** Builds cleanly on existing 0.17 StateView mechanism

### Performance Analysis
- **Without owned-tile cache:** 131K iterations/tick (8 players, 128×128 map) = unacceptable
- **With owned-tile cache:** ~5,600 iterations/tick = sub-millisecond, required from Phase 2
- **Memory:** ~1 MB worst-case for 8 players' visibility sets. Negligible.
- **Day/night transitions:** Potential 15-20ms spike when all radii change simultaneously. Benchmarking recommended.

### Data Structure Feasibility
- `playerViews: Map<sessionId, StateView>` — O(1) lookups, correct
- `visibleTiles: Map<sessionId, Set<number>>` — Memory-efficient, scales well
- **StateView internals:** WeakSet storage is GC-friendly, no memory leaks

### Creature AI Integration
- **Territory barriers:** Unaffected by fog. Wildlife still blocked from owned tiles (server-authoritative)
- **Builder pathfinding:** Uses full server state (correct). No filter dependency.
- **Tick ordering:** Current loop order is safe for visibility as last step
- **Day/night phase:** Already synced globally; fog system uses it for radius modifiers

### Watchtower System Requirements
**Constants needed in shared/src/constants.ts:**
```typescript
WATCHTOWER: {
  COST_WOOD: 15,
  COST_STONE: 10,
  BUILD_TIME_TICKS: 24,
  VISION_RADIUS: 8,
  MAX_PER_PLAYER: 3,

```

- **BuilderAI changes:** Add third buildMode case. Straightforward switch extension.
- **Cost deduction:** 15W/10S per watchtower (different from farms)
- **Max validation:** Requires structure count check. Owned-tile cache mitigates full-map scans.

### Edge Tile Detection Algorithm
- **Cardinal-only edge detection:** Correct. Generates more edge tiles, gives generous visibility.
- **HQ interior exclusion:** All 25 HQ tiles are owned; interior tiles have 4 owned neighbors → not edge. Correct.
- **Map boundaries:** Missing neighbor due to out-of-bounds → treated as edge. Correct.
- **Ring territories:** Interior edges of rings generate inward vision. Correct.

### Client Integration — Critical Gap
**Tile access pattern is breaking change:**
- Current: `state.tiles.at(y * mapWidth + x)`
- With StateView: Filtered tiles → index-based access returns wrong tile
- **Mitigation:** ExploredTileCache provides coordinate-based lookup. GridRenderer already uses per-tile `x`/`y` fields (safe). Audit CreatureRenderer, InputHandler for index-based accesses.

### ExploredTileCache Race Condition
**Race scenario:** Server filters tile removal, client receives DELETE before last state mutation.
- **Acceptable:** One tick of staleness (250ms) is unnoticeable. "Explored" state shows last-known data by design.
- **Recommendation:** Cache `structureType` from day one (for future silhouettes feature)

### User Decision Compatibility

1. **Destructible Watchtowers:** Natural fallout of `tickVisibility()`. Vision loss automatic when structure is deleted.
2. **Shared Alliance Vision:** Multiply visibility sets by alliance size. 4-player alliance = 4× view.add churn. Benchmarking required before 8-player test.
3. **Explored Tile Silhouettes:** Client-side caching of `structureType`. Shows last-known structures (fog semantics). No server change beyond standard StateView filtering.
